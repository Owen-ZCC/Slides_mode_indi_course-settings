## 一、v1.0 方案问题分析

### 1.1 现有方案的问题

| 问题 | 说明 | 影响 |
|------|------|------|
| **选项过多** | 6个入口卡片，用户需要理解每个的区别 | 认知负担重，选择困难 |
| **概念重叠** | 「日常对话」「自由对话」「场景对话」边界模糊 | 用户困惑：我该选哪个？ |
| **层级过深** | 点击后还要再选子类型（基础/深度、朗读/演讲） | 操作路径长，效率低 |
| **命名抽象** | 「任务对话」「日常对话」不够直观 | 不符合用户心智模型 |
| **口语中心冗余** | 作为独立入口价值不大，增加复杂度 | 功能重复，可合并 |

### 1.2 用户真实需求分析

从用户（教师）视角，他们的核心诉求是：

> **「我想让学生练习英语口语，有什么方式？」**

用户关心的是**练习形式**，而非技术实现上的分类。重新梳理：

| 用户想要的 | 对应原功能 | 用户心智 |
|------------|------------|----------|
| 让学生**读一段文章** | 短文朗读 | 跟读/朗读 |
| 让学生**做个小演讲** | 个人演讲 | 演讲/表达 |
| 让学生**和AI聊天** | 基础对话、深度对话、自由对话 | 对话/聊天 |
| 让学生**完成一个情景任务** | 场景对话 | 情景模拟 |
| 让学生**和有趣的角色互动** | 角色对话 | 趣味互动 |

## 二、优化方案：3+1 结构

### 2.1 核心设计理念

**「少即是多」**—— 参考 Notion 的信息架构方式：
- 减少一级选项数量，降低决策成本
- 将相似功能合并，通过配置项区分
- 保持扩展性，不丢失原有能力

### 2.2 新方案：3 个核心入口 + 1 个扩展入口

```
┌─────────────────────────────────────────┐
│  添加英语口语                      [<]   │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────────────────────────────┐   │
│  │  🎯  情景练习                    │   │
│  │  在真实场景中完成对话任务         │   │
│  │  餐厅点餐、问路、购物...          │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │  💬  自由对话                    │   │
│  │  与AI或虚拟角色自由聊天          │   │
│  │  日常话题、深度探讨、角色扮演     │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │  📖  朗读演讲                    │   │
│  │  朗读短文或进行主题演讲          │   │
│  │  AI生成练习材料                  │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │  📚  更多模板                    │   │
│  │  浏览预设的口语练习模板          │   │
│  └─────────────────────────────────┘   │
│                                         │
└─────────────────────────────────────────┘
```

### 2.3 功能映射关系

| 新入口 | 包含原功能 | 合并逻辑 |
|--------|------------|----------|
| **情景练习** | 场景对话 | 独立保留，核心差异化功能 |
| **自由对话** | 基础对话 + 深度对话 + 角色对话 + 自由对话 | 都是「对话」，通过配置区分 |
| **朗读演讲** | 短文朗读 + 个人演讲 | 都是「单向输出」，通过配置区分 |
| **更多模板** | 口语中心 | 简化为文字链接，非主要入口 |

## 三、详细设计

### 3.1 情景练习（Scene Practice）

**定位**：目标导向的情景模拟，有明确的任务完成形态

**为什么独立**：
- 与其他对话形式有本质区别：有**明确任务目标**
- 用户心智清晰：「让学生模拟真实场景」
- 是产品差异化亮点，值得突出

**配置页设计**：

```

├─────────────────────────────────────────┤
│                                         │

│                                         │

│  │ 🍔 餐饮美食                  ▼   │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌────────┐ ┌────────┐ ┌────────┐     │
```

**场景分类**（保持原有 30 个场景）：

| 大类 | 子场景 |
|------|--------|

| 校园生活 | 课堂互动、图书馆借书、食堂点餐 |
| 日常交际 | 询问路线、天气聊天、自我介绍 |
| 节日活动 | 节日祝福、生日派对 |
| 就医安全 | 就医咨询、药店买药 |

### 3.2 自由对话（Free Talk）

**定位**：开放式对话练习，无固定任务目标

**合并逻辑**：
- 基础对话、深度对话、角色对话、自由对话的**本质都是对话**
- 区别仅在于：话题深度、是否有角色设定
- 通过**对话模式**配置项统一管理

**配置页设计**：

**对话模式说明**：

| 深度探讨 | 深度对话 | 探讨抽象话题，层层追问 |
| 角色扮演 | 角色对话 | 代入角色性格和经历对话 |

## 七、类型定义（简化版）

```typescript
// 口语练习大类
export type SpeakingCategory = 'scene' | 'talk' | 'read';

// 对话模式
export type TalkMode = 'casual' | 'deep' | 'roleplay';

// 朗读类型
export type ReadType = 'reading' | 'speech';

// 口语练习配置
export interface SpeakingConfig {
  category: SpeakingCategory;

  // 情景练习配置
  scene?: {
    categoryId: string;
    sceneId: string;
  };

  // 自由对话配置
  talk?: {
    mode: TalkMode;
    roleId?: string;      // 角色扮演时的角色ID
    topic?: string;       // 话题引导
  };

  // 朗读演讲配置
  read?: {
    type: ReadType;
    topic: string;
  };
}
```

## 八、设计原则总结

1. **用户心智优先**：按用户理解的方式组织功能，而非技术实现
2. **渐进式披露**：核心功能前置，高级选项按需展开
3. **减少决策成本**：3 个核心选项，一眼看懂
4. **保持灵活性**：通过配置项保留全部能力，不丢失功能
5. **智能辅助**：推荐系统降低选择门槛
